# Contributing

Thank you for your interest in contributing to this project! Contributions of all kinds are welcome, including bug fixes, new features, documentation improvements, and examples.

## Getting Started

1. Fork the repository.
2. Clone your fork:

   ```bash
   git clone https://github.com/your-username/project-name.git
   ```

3. Create a new branch for your work:

   ```bash
   git checkout -b feature/my-feature
   ```

4. Make your changes and commit them with clear messages.

5. Push your branch:

   ```bash
   git push origin feature/my-feature
   ```

6. Open a Pull Request to the `main` branch.

## Branch Naming Conventions

Branches should be named using a **type prefix** followed by a short, descriptive name for the change. This helps keep the repository organized and makes it easier to understand the purpose of each branch.

Format:

```
type/short-description
```

Examples:

```
feature/editor
bug/viewport-gimbal-lock
feature/particle-system
bug/crash-on-startup
```

Common branch types:

- `feature/` – New functionality or improvements
- `bug/` – Bug fixes
- `docs/` – Documentation changes
- `refactor/` – Code restructuring without changing behavior
- `test/` – Adding or improving tests
- `chore/` – Maintenance tasks (build scripts, dependencies, etc.)

Guidelines:

- Use **lowercase** letters.
- Separate words with **hyphens**.
- Keep names **short but descriptive**.
- Focus each branch on **one specific change**.

## Coding Guidelines

- Keep code **simple and readable**.
- Follow the existing **code style and conventions** in the repository.
- Write **clear commit messages** describing what changed and why.
- Add comments when the code is not immediately obvious.

## Reporting Bugs

If you find a bug, please open an issue and include:

- A clear description of the problem
- Steps to reproduce it
- Expected vs actual behavior
- Screenshots or logs if helpful

## Feature Requests

Feature suggestions are welcome. Please open an issue and describe:

- The problem your feature solves
- A short description of the proposed solution
- Possible alternatives (if any)

## Pull Request Guidelines

Before submitting a PR:

- Ensure the project builds successfully.
- Keep PRs **focused and small** when possible.
- Update documentation if your changes affect usage.

## Code of Conduct

Please be respectful and constructive when interacting with other contributors.

## Questions

If you have questions, feel free to open a discussion or issue.
